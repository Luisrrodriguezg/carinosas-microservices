Carinosas Microservices

Criminal Investigation Case Management System — Technical Documentation

1. Solution Overview

Carinosas is a microservices-based case management system designed for criminal investigation workflows. It manages cases, people involved (victims, suspects, witnesses), evidence with chain of custody, and tasks assigned to investigation team members.

The system follows a loosely-coupled event-driven architecture: services own their data independently and communicate through Apache Kafka events, eliminating synchronous inter-service calls.

Layered Architecture per Service

Each microservice follows the same internal layered structure:

Controller (REST endpoints) ↓ receives HTTP request, validates input via Jakarta Validation Service (Business logic) ↓ orchestrates operations, applies domain rules Repository (Data access) ↓ Spring Data JPA interface extending JpaRepository Domain (JPA Entities + Enums) ↓ maps to PostgreSQL tables via Hibernate (ddl-auto: update) Database (PostgreSQL — one database per service)

Supporting packages: dto/ (request/response records), exceptions/ (custom exceptions + global handler), config/ (Spring configuration beans), event/ (Kafka event classes).

4. Security Architecture

4.1 Authentication — Keycloak + OAuth 2.0

Flow: The frontend authenticates users against Keycloak using the Resource Owner Password Credentials grant. Keycloak returns a signed JWT access token containing the user's realm roles. Every subsequent API request includes this token in the Authorization: Bearer <token> header.

Frontend Keycloak API Gateway Microservice │ │ │ │ │── POST /token ──────────▶│ │ │ │ (username + password) │ │ │ │◀── JWT access_token ─────│ │ │ │ │ │ │ │── GET /api/cases ────────────────────────────────────────▶│ │ │ Authorization: Bearer JWT │ │ │ │ │── validate JWT ───────▶│ │ │◀── fetch JWKS ───────────────│ (signature check) │ │ │── public keys ──────────────▶│ │ │ │ │── extract roles ──────▶│ │ │ │── check RBAC ─────────▶│ │ │ │── forward request ────▶│ │◀──────────────────────────────────────────────────────────│◀── response ──────────│

Keycloak configuration:

  • Realm: carinosas
  • Client: carinosas-client (public, direct access grants enabled)
  • JWT validation: Gateway fetches public keys from Keycloak JWKS endpoint to verify token signatures

4.2 Authorization — Role-Based Access Control (RBAC)

The API Gateway extracts roles from the JWT realm_access.roles claim and enforces access rules before forwarding requests to downstream services. Individual microservices do not perform authorization — they trust the gateway.

4.3 Gateway Security Implementation

Implemented in SecurityConfig.java using Spring Security's reactive ServerHttpSecurity:

  • CSRF: Disabled (stateless JWT-based API)
  • CORS: Configured to allow all origins, methods, and headers
  • JWT converter: Custom ReactiveJwtAuthenticationConverter extracts Keycloak's realm_access.roles and maps them to Spring Security ROLE_* authorities
  • Actuator endpoints: Unauthenticated (health checks)
  • OPTIONS preflight: Permitted without authentication

5. Event-Driven Communication — Apache Kafka

5.1 Problem

Each microservice owns its own database. When a task references a person and an evidence item, those services have no knowledge of the relationship. Querying "which tasks are assigned to this person?" would require synchronous cross-service REST calls — coupling services at runtime.

5.2 Solution — Event Sourcing via Kafka

Task-service publishes events to a task-events Kafka topic. Person-service and evidence-service consume these events and persist local projection tables, enabling queries without inter-service calls.

  task-service                    Kafka                     person-service
       │                           │                              │
       │── save Task to DB         │                              │
       │── publish TaskEvent ────▶ │ topic: task-events           │
       │                           │ ─────────────────────────────▶ consume event
       │                           │                              │── upsert task_assignments table
       │                           │                              │
       │                           │                        evidence-service
       │                           │                              │
       │                           │ ─────────────────────────────▶ consume event
       │                           │                              │── upsert task_evidence_links table

5.3 Event Structure

FieldTypeDescription
eventTypeStringTASK_CREATED, TASK_UPDATED, or TASK_DELETED
taskIdUUIDUnique identifier of the task
titleStringTask title (denormalized for consumer queries)
statusStringPENDING, IN_PROGRESS, COMPLETED, OVERDUE, CANCELLED
priorityStringLOW, MEDIUM, HIGH, CRITICAL

assignedPersonIdUUIDPerson assigned (nullable)
evidenceIdUUIDEvidence linked (nullable)

5.4 Consumer Behavior

Eventperson-serviceevidence-service
TASK_CREATEDInsert row in task_assignments if assignedPersonId is setInsert row in task_evidence_links if evidenceId is set
TASK_UPDATEDUpsert: update existing row or insert new one. Remove if assignedPersonId is now nullUpsert: update existing row or insert new one. Remove if evidenceId is now null
TASK_DELETEDDelete row by taskIdDelete row by taskId

5.5 Kafka Configuration

  • Mode: KRaft (no Zookeeper required)
  • Topic: task-events (auto-created)
  • Message key: taskId (ensures per-task ordering)
  • Serialization: Custom JSON serializer/deserializer using Jackson 3.x (compatible with Spring Boot 4.x)
  • Consumer groups: people-service and evidence-service (each receives all events independently)
  • Offset reset: earliest (consumers read from beginning on first connect)

6.2 Health Checks & Startup Order

Docker Compose uses health checks to enforce startup order via depends_on: condition: service_healthy:

  • PostgreSQL: pg_isready
  • Kafka: nc -z kafka 9092 (TCP port check)
  • Keycloak: GET /health/ready (HTTP 200)
  • Microservices: GET /actuator/health (Spring Boot Actuator)

6.3 Database Isolation

A single PostgreSQL instance hosts four separate databases, one per service. Each service connects only to its own database — there are no cross-database queries or shared tables.

6.4 Dockerfile Pattern

All services use the same minimal Dockerfile:

FROM eclipse-temurin:21-jre-alpine WORKDIR /app COPY target/*.jar app.jar EXPOSE {port} ENTRYPOINT ["java", "-jar", "app.jar"]

JARs are pre-built with Maven (./mvnw clean package) before docker compose up --build.

Key relationships:

  • Case is the central aggregate — People, Evidence, and Tasks reference it via caseId
  • Task references a Person (assignedPersonId) and Evidence (evidenceId) by UUID
  • All cross-service references are UUID-only (no foreign key constraints — different databases)
  • task_assignments and task_evidence_links are materialized views populated by Kafka events
